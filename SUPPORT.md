# Support

## What to use where

Use the GitHub surfaces based on the type of help you need:

- **Bug reports**: open a GitHub issue with the bug template
- **Feature ideas**: open a GitHub issue with the feature template
- **Pull requests**: use the PR template and include validation commands
- **Security issues**: do not file a public issue; follow `SECURITY.md`

## What makes support requests actionable

The most useful reports include:

- the tool involved: `AgentCI`, `TracePack`, `FailMap`, `PackSlice`, or root automation
- exact commands run
- sample input artifact paths
- expected vs actual output
- failing JSON field, manifest shape, or report difference
- environment details when relevant

## Before opening an issue

Please check:

- `README.md` for the repo-level workflow
- `docs/automation.md` for the end-to-end artifact story
- package-level READMEs under `projects/*`
- existing issues for duplicates

## Best-effort support boundaries

This repository is maintained as an OSS toolkit, so support is best-effort.

Requests that are easiest to help with are:

- reproducible bugs
- docs gaps tied to runnable commands
- artifact contract inconsistencies
- focused feature proposals that strengthen the eval toolchain
